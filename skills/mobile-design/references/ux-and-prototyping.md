## Research framing

- Identify the user segment, context of use, and success criteria before proposing flows.
- Ask what the user is trying to accomplish, what blocks them today, and what would make the task feel faster or safer.
- Distinguish between a user problem and a stakeholder preference.

## User journeys

- Map the user goal from entry trigger to outcome.
- Call out decision points, uncertainties, and high-friction moments.
- Include alternate paths for empty states, errors, cancellations, and returns.
- Use journey maps when the problem spans multiple screens or time gaps.

## Information architecture

- Group content by user intent, not by internal team ownership.
- Keep labels concrete and easy to scan on small screens.
- Reduce depth where possible; mobile users should not traverse many stacked choices just to complete a basic action.
- Validate that naming, grouping, and sequence match the user's mental model.

## Wireframes and mockups

- Use low-fidelity wireframes when the structure is still changing.
- Use high-fidelity mockups when hierarchy, branding, and interaction nuance need evaluation.
- For paper-style output, keep annotations simple and focus on task flow and layout blocks.
- For digital mockup specs, define key components, copy, spacing rhythm, and state changes.

## Prototype strategy

- Prototype only the interactions that carry uncertainty or risk.
- Use simple click-through flows for IA validation.
- Use richer interactive prototypes for gestures, transitions, or multi-step decision flows.
- Keep prototypes narrow enough that feedback targets the right question.

## Validation and iteration

- Test with representative tasks, not abstract opinions.
- Ask users to narrate expectations before they tap so mismatched mental models become visible.
- Sort findings into comprehension, trust, efficiency, and delight issues.
- Iterate from the highest-severity friction first instead of polishing already-working details.
